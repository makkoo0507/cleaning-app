## 前提条件

以下の2つのみ Mac にインストールされていれば OK。

| ツール | 確認コマンド | 備考 |
| --- | --- | --- |
| Docker Desktop | `docker -v` | 起動しておくこと |
| Node.js v20+ | `node -v` | supabase CLI の実行にのみ使用 |

> **方針: Mac には何もインストールしない**
>
> - `create-next-app`（初期 scaffold）→ コンテナ内で実行
> - `npm install`（依存パッケージ）→ コンテナ内で実行・named volume に保持
> - Mac の Node.js は Supabase CLI の実行専用（Supabase サービスは Docker 管理）

## 初回セットアップ

```bash
# 1. リポジトリをクローン（GitHub 作成後）
git clone <リポジトリURL>
cd cleaning-app

# 2. セットアップ
#    - docker-compose build でコンテナ内に node_modules を作成
#    - supabase start で Supabase を Docker で起動
make setup
```

`make setup` が完了すると、ターミナルに以下のような出力が表示される。

```
API URL: http://localhost:54321
anon key: eyJhbGci...
service_role_key: eyJhbGci...
```

```bash
# 3. .env.local に値を貼り付け
#    NEXT_PUBLIC_SUPABASE_ANON_KEY=（anon key）
#    SUPABASE_SERVICE_ROLE_KEY=（service_role_key）
vi .env.local  # または VS Code で編集
```

```bash
# 4. Next.js を Docker で起動
docker-compose up
```

ブラウザで http://localhost:3000 を開いて確認。

## 毎日の開発起動手順

```bash
# Docker Desktop を起動してから
make start
```

- Next.js: http://localhost:3000
- Supabase Studio: http://localhost:54323
- 公開URL(ngrok): https://<固定ドメイン>/ … LIFF 実機テスト用（`make start` で自動起動）

> `make start` は Supabase + Next.js(-d) + ngrok をすべてバックグラウンド起動する。
> ngrok のログは `make tunnel-logs`、状況は http://localhost:4040。
> ngrok だけ起動: `make tunnel`。

### LINE Mini App（LIFF）の実機テスト

LIFF は HTTPS 必須で、スマホからは `localhost` に到達できないため ngrok 経由で行う。

1. ngrok 固定ドメインを取得し、LIFF アプリ（単一）の**エンドポイントを `https://<ngrok>/liff`** に設定。
2. `.env.local` の `NEXT_PUBLIC_LIFF_ID` にそのテスト LIFF ID を設定。
   - **`.env.local` 変更後は `docker compose up -d --force-recreate app`**（`restart` では
     `NEXT_PUBLIC_` 系が反映されない）。
3. 招待リンク `https://liff.line.me/{LIFF_ID}/invite?token=...` を**LINE アプリ内**で開く。
4. 紐付け後の画面: `https://liff.line.me/{LIFF_ID}/cleaner/schedules` / `.../owner/schedules`。

> Mini App はデータ取得もサーバー経由のため、ローカル Supabase のままで実機表示できる。

## マイグレーション追加手順

```bash
# 新しいマイグレーションファイルを作成
npx supabase migration new <ファイル名>
# → supabase/migrations/YYYYMMDD_<ファイル名>.sql が作成される

# ローカル DB に適用
npx supabase db reset
```

### 本番への反映（重要）

本番（Supabase Cloud）は **git push では自動反映されない**。マイグレーション追加後は手動で push する。

```bash
npx supabase db push   # supabase/migrations/ を本番に適用
```

> 注意: `seed.sql` は `db push` では反映されない（`db reset` 時のローカルのみ）。
> Vercel（Next.js）は main への push で自動デプロイされるが、Supabase は別途 `db push` が必要。

## ファイル構成

```text
cleaning-app/
├── Makefile                    # 開発コマンド集
├── Dockerfile.dev              # Next.js 開発用コンテナ
├── docker-compose.yml          # Next.js サービス定義
├── .env.local.example          # 環境変数テンプレート
├── .env.local                  # 実際の環境変数（Git 管理外）
├── supabase/
│   ├── config.toml             # Supabase ローカル設定
│   ├── seed.sql                          # ローカル用シード（テスト業者+管理者）
│   └── migrations/
│       ├── 20260620000000_init.sql            # テーブル定義
│       ├── 20260620000001_rls.sql             # RLS ポリシー
│       ├── 20260620000002_auth_hook.sql       # JWT カスタムクレーム
│       ├── 20260621000000_fix_auth_hook.sql   # auth hook を security definer 化
│       ├── 20260621000001_grants_and_profile_policies.sql  # authenticated GRANT + プロフィール系ポリシー
│       ├── 20260621000002_company_slug.sql（リネーム不可・マイグレーション履歴）    # 業者別ログインURL用 slug
│       └── 20260621000003_service_role_grants.sql         # service_role GRANT
└── next/                       # Next.js アプリ（Vercel の Root Directory）
```

> **node_modules について（2026-06-21 変更）**
> 以前は named volume に隔離していたが、Mac の VS Code で型補完・Lint を効かせるため
> `docker-compose.yml` で `./next` を bind マウントする方式に変更した。
> パッケージ追加は **コンテナ内で実行**すること（`docker compose exec app npm install <pkg>`）。
> Mac 側で直接 `npm install` するとネイティブモジュールが mac 用に上書きされ動かなくなる場合がある。

> **ローカルのテストログイン**（`supabase/seed.sql` で投入。`make reset` で再適用）
> - URL: `/acme/login`（業者別ログイン。汎用 `/login` は廃止。`/` から業者ID `acme` を入力しても可）
> - メール: `admin@example.com` / パスワード: `password123`
