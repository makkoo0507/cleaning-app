# ベンダー向け：定期リマインドの有効化手順

> 対象：システム運営（ベンダー）担当者
> 業者向けの操作（送信先・タイミングの設定）はアプリ内マニュアル `/manual/reminder` を参照。
> 本書は、自動送信を動かすためのサーバー側の初期設定（本番 Supabase）をまとめたもの。

## 仕組み（概要）

```
pg_cron（毎日 指定時刻にトリガー）
   ↓ net.http_post()（pg_net）
Edge Function「daily-reminder」（kind=prev_day / same_day）
   ↓ 業者ごとの設定に従い対象案件を抽出
LINE Messaging API へ push（各業者のチャネルトークンで送信）
```

- 前日リマインド（kind=prev_day）：翌日の `scheduled` 案件を前日夜に送信
- 当日リマインド（kind=same_day）：当日の `scheduled` 案件を当日朝に送信
- 送信先（清掃者／オーナー）× タイミング（前日／当日）は業者ごとの設定（`contractors`）に従う
- `reminder_logs(job_id, kind)` のユニーク制約で重複送信を防止

## 手順（本番 Supabase で初回のみ）

### 1. 拡張機能を有効化

Supabase ダッシュボード → **Database → Extensions** で以下を有効化：

- `pg_cron`（これで `cron.job` が作られ、Cron 登録が可能になる）
- `pg_net`（Edge Function を HTTP で呼ぶため）

> 先に有効化しないと、Cron ジョブ作成時に `relation "cron.job" does not exist` エラーになる。

### 2. Edge Function をデプロイ

```bash
supabase functions deploy daily-reminder
```

### 3. Cron ジョブを2本登録（ダッシュボード Cron UI）

**Integrations → Cron**（または **Database → Cron Jobs**）→ **Create job**

| 項目 | 前日リマインド | 当日リマインド |
|---|---|---|
| Name | `reminder-prev-day` | `reminder-same-day` |
| Schedule | `0 11 * * *`（20:00 JST） | `0 23 * * *`（翌 8:00 JST） |
| Type | Supabase Edge Function → `daily-reminder` | 同左 |
| Method | `POST` | `POST` |
| Body | `{"kind":"prev_day"}` | `{"kind":"same_day"}` |

- スケジュールは **UTC 指定**（20:00 JST = 11:00 UTC、8:00 JST = 23:00 UTC）。

> ⚠️ **既知のハマりどころ（重要）**
> Cron UI で作成すると、ジョブの `headers` が空 `{}` のまま（Authorization が付かない）になることがある。
> その場合 Edge Function が **401（Invalid JWT / Unauthorized）で実行されず**、`cron.job_run_details` は
> `succeeded`／`net._http_response` は `status_code = null` という分かりにくい状態になる。
> また Cron UI の **Timeout は最大 5000ms** までしか入れられない。
> → **確実なのは下記「SQL で登録」**（`jsonb_build_object` で Authorization を明示、timeout も自由）。
> 使うキーは **必ず Legacy API keys の `service_role`（`eyJ...` で始まる JWT）**。
> 新方式の `sb_secret_...` / `sb_publishable_...` は JWT ではないため 401 になる。

#### SQL で登録する場合（代替）

```sql
ALTER DATABASE postgres SET app.supabase_functions_url
  TO 'https://<project-ref>.supabase.co/functions/v1';
ALTER DATABASE postgres SET app.service_role_key TO '<service_role_key>';

SELECT cron.schedule('reminder-prev-day', '0 11 * * *', $$
  SELECT net.http_post(
    url     := current_setting('app.supabase_functions_url') || '/daily-reminder',
    headers := jsonb_build_object(
                 'Authorization', 'Bearer ' || current_setting('app.service_role_key'),
                 'Content-Type',  'application/json'),
    body    := '{"kind":"prev_day"}'::jsonb); $$);

SELECT cron.schedule('reminder-same-day', '0 23 * * *', $$
  SELECT net.http_post(
    url     := current_setting('app.supabase_functions_url') || '/daily-reminder',
    headers := jsonb_build_object(
                 'Authorization', 'Bearer ' || current_setting('app.service_role_key'),
                 'Content-Type',  'application/json'),
    body    := '{"kind":"same_day"}'::jsonb); $$);
```

設定値の確認場所：

- `service_role_key`：Project Settings → API Keys → **Legacy API keys** → `service_role`（`eyJ...`）
- `FUNCTIONS_URL`：通常 `https://<project-ref>.supabase.co/functions/v1`

#### 実績のある登録方法（キー直書き・timeout 30秒）

`current_setting` を使わず、キーを直接埋め込む形（今回の本番で動作確認済み）：

```sql
select cron.unschedule('reminder-prev-day');
select cron.unschedule('reminder-same-day');

select cron.schedule('reminder-prev-day', '0 11 * * *', $$
  select net.http_post(
    url := 'https://<project-ref>.supabase.co/functions/v1/daily-reminder',
    headers := jsonb_build_object(
      'Authorization', 'Bearer <eyJ...service_role>',
      'Content-Type', 'application/json'),
    body := '{"kind":"prev_day"}'::jsonb,
    timeout_milliseconds := 30000
  ); $$);

select cron.schedule('reminder-same-day', '0 23 * * *', $$
  select net.http_post(
    url := 'https://<project-ref>.supabase.co/functions/v1/daily-reminder',
    headers := jsonb_build_object(
      'Authorization', 'Bearer <eyJ...service_role>',
      'Content-Type', 'application/json'),
    body := '{"kind":"same_day"}'::jsonb,
    timeout_milliseconds := 30000
  ); $$);
```

> service_role キーが平文で入るため、実行後は SQL Editor の履歴/タブから消すこと。

#### 動作確認（時刻を待たずに）

```sql
-- 1) 手動で1回叩く（上と同じ http_post を単体実行）
-- 2) 数秒後、応答を確認
select status_code, content, created from net._http_response order by created desc limit 3;
```

- `status_code = 200` ＋ `content` に `ok: N sent (...)` が出れば成功。
- `status_code = null` → ヘッダー不足（401）かタイムアウト。`status_code = 401` → キーが Legacy の `eyJ...` でない。
- cron の発火履歴：`select * from cron.job_run_details order by start_time desc limit 10;`（時刻は UTC）。

## 動作確認・運用

- 手動実行（送信ロジックの確認）：
  ```bash
  curl -X POST "{FUNCTIONS_URL}/daily-reminder" \
    -H "Authorization: Bearer <service_role_key>" \
    -H "Content-Type: application/json" \
    -d '{"kind":"prev_day"}'
  ```
  レスポンスの `ok: N sent (...)` で送信件数を確認できる。
- 再テストで同じ案件に再送したい場合は、`reminder_logs` の該当行を削除する。
- 登録済みジョブの確認：`select jobname, schedule, active from cron.job;`
- ジョブ削除：`select cron.unschedule('reminder-prev-day');`

## ローカル開発について

- ローカル Supabase にも pg_cron / pg_net は同梱されているが、ダッシュボードの Cron UI が
  生成する URL がプレースホルダ（`default.supabase.1`）になり、スケジュール実行は関数に届かない。
- ローカルでは上記 curl の**手動実行**で関数の動作確認に留めるのが現実的。
