## 1. 技術スタック

| 層 | 技術 | 選定理由 |
| --- | --- | --- |
| フロントエンド（管理 Web） | Next.js (App Router) | SSR・API Routes を同一リポジトリで管理できる |
| フロントエンド（LINE Mini App） | Next.js + LIFF SDK | 管理 Web と同一フレームワーク・同一リポジトリで開発可能 |
| バックエンド / DB | Supabase | PostgreSQL、認証、リアルタイム購読、ストレージをオールインワンで提供。小規模向けの無料枠あり |
| 認証（業者） | Supabase Auth (Email / Password) | — |
| 認証（清掃者・オーナー） | LINE Login (LIFF) → Supabase セッション発行 | — |
| LINE 通知 | LINE Messaging API (Push Message) | — |
| スケジューラー（リマインド） | Supabase Edge Functions + pg_cron | 前日リマインドを cron で実行 |
| ホスティング | Vercel | Next.js との親和性が高く、無料枠で小規模運用可能 |

## 5. 環境変数（.env.local）

```env
# Supabase
NEXT_PUBLIC_SUPABASE_URL=
NEXT_PUBLIC_SUPABASE_ANON_KEY=
SUPABASE_SERVICE_ROLE_KEY=

# LIFF（単一アプリで全役割共有。エンドポイント /liff）
NEXT_PUBLIC_LIFF_ID=
```

> **LIFF は 1 アプリに統一**（旧: 清掃者用/オーナー用の 2 つ → 廃止）。
> 同一エンドポイント `/liff` を 2 アプリで共有すると、launch 時に liffId を特定できず
> ログインが完了しない不具合があったため、単一 LIFF に統一し、役割は
> ユーザーレコード（`users.role`）で判定する。ローカルは ngrok の `/liff`、本番は
> 本番ドメインの `/liff` をエンドポイントに設定する。

> **LINE Messaging API の Token について**
> Channel Access Token と Channel Secret は業者ごとに異なるため、環境変数では管理しない。
> `contractors.line_channel_access_token` / `line_channel_secret` カラムに保存し、通知送信時に DB から取得する。

## 6. LINE 通知実装メモ

### Push Message 送信（サーバーサイド）

Channel Access Token は業者ごとに DB から取得する。

```typescript
// lib/line.ts
import axios from 'axios';

export async function pushMessage(
  channelAccessToken: string, // contractors.line_channel_access_token
  lineUserId: string,
  text: string
) {
  await axios.post(
    'https://api.line.me/v2/bot/message/push',
    {
      to: lineUserId,
      messages: [{ type: 'text', text }],
    },
    {
      headers: {
        Authorization: `Bearer ${channelAccessToken}`,
      },
    }
  );
}
```

### LIFF / Mini App のアーキテクチャ（重要）

ブラウザ（スマホ）は **Supabase に直接アクセスせず、Next サーバー経由のみ**でデータを扱う。
これによりローカル開発（スマホから `localhost:54321` に到達できない）でも実機テストできる。

- **認証**: クライアント（`LiffBootstrap`）は `liff.init`/`login`/`getProfile` で LINE ID を取得し
  `/api/liff/auth` に渡すだけ。サーバーが `verifyOtp` してセッションを **Cookie 発行**する。
  - `role` を渡して「今いる画面の役割」のユーザーでログイン（同一 LINE アカウントが
    清掃者・オーナー両方に紐付いていてもパスで一意に解決。開発テストに有用）。
- **データ取得**: 清掃者/オーナー画面は **Server Component**。`lib/liff-auth.ts` の
  `getLiffUser()` でセッションからユーザーを特定し、本人スコープ（cleaner_id /
  property_members）で取得（admin client + 明示フィルタ）。
- **更新**: 清掃開始/完了/共有メモ更新は `/api/liff/record`（Route Handler）。
- **入口 URL**: LIFF は単一アプリ・エンドポイント `/liff`。
  - 招待: `https://liff.line.me/{LIFF_ID}/invite?token=...`
  - 通知内リンク: `https://liff.line.me/{LIFF_ID}/cleaner/schedules`（または `/owner/schedules`）
  - `liff.line.me/{id}/<path>` は内部的に `/liff?liff.state=<path>` で開くため、`/liff` 入口は
    `liff.state` がある時は役割リダイレクトせず SDK の遷移に任せる。
- **ローカル実機テスト**: ngrok（固定ドメイン）で `localhost:3000` を HTTPS 公開し、
  LIFF エンドポイントをそのドメインの `/liff` に設定。`make start` / `make tunnel` 参照。

> 旧 `lib/liff.ts`（クライアント直叩きの initLiff）や `LiffAuthGuard` は廃止。

## 7. Supabase RLS ポリシー方針

**大原則: 全ポリシーに `contractor_id = (自分の contractor_id)` の条件を必ず付加し、テナント間のデータ漏洩を防ぐ。**

| テーブル | 業者管理者 | 業者社員 | 清掃者 | オーナー |
| --- | --- | --- | --- | --- |
| contractors | 自社のみ読取 | 自社のみ読取 | アクセス不可 | アクセス不可 |
| properties | 自社内 全 CRUD | 自社内 全読取・作成・編集 | 自社内 自案件の物件のみ読取 | 自社内 自物件のみ読取 |
| users | 自社内 全 CRUD | 自分のみ読取・更新 | 自分のみ読取・更新 | 自分のみ読取・更新 |
| jobs | 自社内 全 CRUD | 自社内 全読取・作成・編集 | 自社内 自分がアサインされた案件のみ読取 | 自社内 自物件の案件のみ読取 |
| cleaning_records | 自社内 全 CRUD | 自社内 全読取 | 自社内 自分の案件への書込・読取 | 自社内 自物件の案件の読取のみ |
