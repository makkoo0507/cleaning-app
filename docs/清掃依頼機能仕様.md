# 清掃依頼機能 仕様書

**作成日:** 2026-06-27  
**ステータス:** 設計確定・実装待ち

---

## 概要

民泊オーナーが LINE Mini App から清掃を依頼できる機能。管理者が依頼を確認し、承認または却下する。承認時に正式な案件としてスケジュールに登録される。

---

## データ設計

### cleaning_requests テーブル（新規）

| カラム | 型 | 説明 |
|---|---|---|
| id | uuid | PK |
| contractor_id | uuid | 業者ID（FK） |
| property_id | uuid | 物件ID（FK） |
| requested_by | uuid | 依頼したオーナーのユーザーID（FK） |
| requested_date | date | 希望清掃日 |
| requested_start_time | time | 希望開始時刻（nullable） |
| note | text | オーナーからのメモ・変更希望（nullable） |
| status | text | `pending` / `approved` / `rejected` |
| rejection_reason | text | 却下理由（nullable、却下時のみ） |
| created_at | timestamp | |

### jobs テーブル（変更）

| カラム | 型 | 説明 |
|---|---|---|
| request_id | uuid | FK → cleaning_requests（nullable） |

- 依頼から作成された案件のみ `request_id` を持つ
- 手動作成の案件は `request_id = null`

---

## 機能仕様

### 1. オーナー側（LINE Mini App）

**場所:** スケジュール確認メニュー内に「依頼する」ボタンを追加

**依頼フォーム:**
- 日付（必須、オーナーが選択）
- 参考情報として以下を表示（変更不可）
  - 物件のデフォルト開始時刻
  - 物件のデフォルト請求金額
- メモ欄（任意）：時間変更希望や特記事項を記入

**送信後:** `cleaning_requests` に `status = pending` で登録

---

### 2. 管理者側（管理 Web アプリ）

#### スケジュール画面への表示

- リスト・週・月の全ビューに依頼（pending）を表示
- **アンバー系の背景色**で通常の案件と視覚的に区別
- 依頼カードに表示する情報：物件名・希望日時・オーナー名・メモ

#### 承認

1. 「承認」ボタンを押す
2. 案件新規作成ページへ遷移（依頼データをプリフィル）
   - 物件、清掃日、開始時刻、請求額・支払い額がデフォルト値でセット
   - 修正可能
3. 保存すると `jobs` レコードが作成され `jobs.request_id` で依頼と紐付け
4. `cleaning_requests.status` が `approved` に更新

#### 却下

1. 「却下」ボタンを押す
2. 理由記入モーダルが開く（任意入力）
3. 送信すると `cleaning_requests.status` が `rejected`、`rejection_reason` に理由を保存
4. オーナーに LINE 通知（理由を含む）

#### ダッシュボード

- 要対応セクションに未対応の依頼（pending）件数を表示（未アサインと同様）

---

## 通知

| タイミング | 送信先 | 内容 |
|---|---|---|
| 依頼受付時 | 管理者（LINE通知 or ダッシュボード） | 新着依頼の通知 |
| 却下時 | 依頼したオーナー | 却下理由を含むLINEメッセージ |
| 承認・案件確定時 | 依頼したオーナー | 既存の案件作成通知と同様 |

## 実装順序（案）

1. DBマイグレーション（`cleaning_requests` テーブル作成・`jobs.request_id` 追加）
2. 管理画面スケジュール表示への依頼の混在（アンバー背景）
3. 承認UI（案件作成ページへの遷移・プリフィル・request_id 紐付け）
4. 却下UI（理由記入モーダル・LINE通知）
5. LINE Mini App 依頼フォーム（オーナー側）
6. ダッシュボード要対応セクションへの依頼件数追加
