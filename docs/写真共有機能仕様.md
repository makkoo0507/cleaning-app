# 写真共有機能 仕様書

**作成日:** 2026-06-28  
**ステータス:** 設計確定・実装待ち

---

## 概要

清掃完了後に清掃者が写真をアップロードし、業者が確認・選別して、オーナーへ LINE で共有する機能。  
現状の「清掃者 → LINE → 業者 → Google Drive → オーナー」という手動フローを置き換える。

---

## 現状フローと課題

```
現状:
1. 清掃者がLINEで写真を業者に送付
2. 業者が必要な写真を端末に保存
3. 業者がGoogle Driveにアップロード
4. Google Drive のURLを手動メッセージでオーナーに共有

課題:
- 4ステップの手動作業が必要
- Google Driveの管理が必要
- LINEを離れてしまう
```

---

## 新フロー

```
【清掃者 / LIFF】
  ├─ 清掃中・完了後どちらでも写真アップ可
  └─ 「完了」ボタン → 管理画面に"完了"表示（オーナーへは通知しない）

【業者 / 管理画面】
  ├─ 案件詳細で全写真を確認
  ├─ オーナーに共有する写真をチェックして選別
  ├─ 共有メモ（既存フィールド）に報告メッセージを記入
  └─ 「オーナーに報告」ボタン →
       LINE メッセージ送信（共有メモ本文 + 「写真を見る」ボタン）

【オーナー / LINE・LIFF】
  └─ 「写真を見る」タップ → LIFF フォトギャラリーページ
       （share_with_owner = true の写真のみ表示）
```

---

## 通知フローの変更

| タイミング | 現状 | 変更後 |
|---|---|---|
| 清掃完了 | オーナーに自動通知 | **通知なし**（管理画面のみ反映） |
| 業者が「報告」ボタンを押す | なし | **オーナーに通知**（本文＋写真リンク） |

### jobs テーブル（変更）

|---|---|---|
| reported_at | timestamptz | オーナーへ報告した日時（nullable） |

## ストレージ設計

### Supabase Storage

- バケット名: `cleaning-images`
- アクセス: プライベート（RLS で制御）

### RLS 方針

| ロール | 権限 |
|---|---|
| 清掃者（cleaner） | アサインされている案件の画像をアップロード・閲覧 |
| 業者スタッフ（contractor_admin / contractor_vendor） | 自社案件の全画像を閲覧・削除 |
| オーナー（contact） | share_with_owner = true の自物件画像のみ閲覧 |

## 費用・パフォーマンス対策

### ストレージ費用（Freeプランのまま運用可能）

60日で自動削除するため、常時の保存容量は直近60日分のみ。

常時保存量 = 月間案件数 × 12MB（30枚×400KB） × 2ヶ月（60日分）

1案件あたりの上限: 30枚（超過時はアップロード不可）

| 月間案件数 | 1案件あたり | 常時保存量 | 状況 |
|---|---|---|---|
| 月10案件 | 30枚 × 400KB = 12MB | 約240MB | 余裕あり |
| 月20案件 | 30枚 × 400KB = 12MB | 約480MB | 余裕あり |
| 月30案件 | 30枚 × 400KB = 12MB | 約720MB | 余裕あり |
| 月40案件 | 30枚 × 400KB = 12MB | 約960MB | ほぼ上限 |
| **月43案件** | 30枚 × 400KB = 12MB | **約1,032MB** | **Free上限（1GB）到達** |

月43案件を超えたタイミングで Proプラン（$25/月・100GB）への移行が必要。

### クライアント側圧縮（Proプラン不要にするため）

アップロード前にブラウザ（Canvas API）で変換:
- 長辺: 最大 1200px
- 形式: JPEG
- 品質: 80%
- 結果: 1枚あたり 300〜500KB（元の iPhoneカメラ写真は 5〜8MB）

### ページ表示速度

- 一覧は CSS でサムネイルサイズに縮小表示（別ファイル不要）
- タップで原寸表示
- 遅延ロード（lazy loading）を適用

---

## 保存期間・バックアップ

### 自動削除

- 保存期間: **60日**
- 削除方法: 既存のリマインダーと同様の Edge Function + pg_cron で日次実行
- 削除対象: `created_at < now() - interval '60 days'` の cleaning_images
  - Supabase Storage から削除
  - DB レコードから削除

### バックアップ（削除前にダウンロード）

削除7日前から管理画面の案件詳細にバナー表示:
「この案件の写真は XX日後に削除されます。必要な場合はダウンロードしてください。」

### ダウンロード機能

| 機能 | 業者（管理画面） | オーナー（LIFF） |
|---|---|---|
| 個別ダウンロード | ✓ | ✓ |
| まとめて ZIP ダウンロード | ✓（全写真） | ✓（共有写真のみ） |

ZIP 生成: クライアント側で JSZip を使用（サーバー負荷なし）

## 並行開発ガイド（3名体制）

### 事前合意事項（全員が作業開始前に確認すること）

#### Storage パス形式

```text
{contractor_id}/{job_id}/{timestamp}_{random}.jpg
```

#### オーナー向け写真閲覧URL

```text
/liff/owner/jobs/{job_id}
```

既存のオーナー案件詳細ページに写真セクションを追加する。
「オーナーに報告」ボタンが送るLINEメッセージの「写真を見る」リンクもこのURLを使う。

#### 画像取得クエリの基本形（各担当ページで共通して使う）

```typescript
// 業者・管理者用（全写真）
supabase.from("cleaning_images").select("*").eq("job_id", jobId).order("created_at")

// オーナー用（共有写真のみ）
supabase.from("cleaning_images").select("*").eq("job_id", jobId).eq("share_with_owner", true).order("created_at")

// 清掃者用（アサインされた案件の全写真）
supabase.from("cleaning_images").select("*").eq("job_id", jobId).order("created_at")
```

## 写真の削除仕様

| 操作者 | 削除できる写真                          |
|--------|-----------------------------------------|
| 清掃者 | `share_with_owner = false` の写真のみ   |
| 管理者 | すべての写真（いつでも）                |

`share_with_owner = true` になった時点で清掃者は削除不可となる。管理者による共有指定が自動的に保護ロックとして機能するため、別途ロック操作は不要。
